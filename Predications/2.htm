Étape 1 - Définir le schéma

Pour les besoins de ce didacticiel, vous construirez à partir d'un schéma qui définit l'inventaire d'une boutique en ligne qui vend une variété d'appareils électroniques.

Traditionnellement, le modèle de modèle Entité-Attribut-Valeur (EAV) était utilisé pour permettre aux clients de comparer les caractéristiques des produits.

Cependant, avec le type de données JSON, ce cas d'utilisation peut être abordé différemment.

La base de données sera nommée e_storeet aura trois tables nommées brands, categorieset productsrespectivement.

Créez la e_storebase de données :

CREATE DATABASE IF NOT EXISTS `e_store`

DEFAULT CHARACTER SET utf8

DEFAULT COLLATE utf8_general_ci;

 

SET default_storage_engine = INNODB;

Copie

Les tables brandset categoriesauront chacune un idet un namechamp.

Créez le brandstableau :

CREATE TABLE `e_store`.`brands`(

        `id` INT UNSIGNED NOT NULL auto_increment ,

        `name` VARCHAR(250) NOT NULL ,

        PRIMARY KEY(`id`)

);

Copie

Créez le categoriestableau :

CREATE TABLE `e_store`.`categories`(

        `id` INT UNSIGNED NOT NULL auto_increment ,

        `name` VARCHAR(250) NOT NULL ,

        PRIMARY KEY(`id`)

);

Ensuite, créez une productstable avec les champs id, name, brand_id, category_idet :attributes

CREATE TABLE `e_store`.`products`(

        `id` INT UNSIGNED NOT NULL AUTO_INCREMENT ,

        `name` VARCHAR(250) NOT NULL ,

        `brand_id` INT UNSIGNED NOT NULL ,

        `category_id` INT UNSIGNED NOT NULL ,

        `attributes` JSON NOT NULL ,

        PRIMARY KEY(`id`) ,

        INDEX `CATEGORY_ID`(`category_id` ASC) ,

        INDEX `BRAND_ID`(`brand_id` ASC) ,

        CONSTRAINT `brand_id` FOREIGN KEY(`brand_id`) REFERENCES `e_store`.`brands`(`id`) ON DELETE RESTRICT ON UPDATE CASCADE ,

        CONSTRAINT `category_id` FOREIGN KEY(`category_id`) REFERENCES `e_store`.`categories`(`id`) ON DELETE RESTRICT ON UPDATE CASCADE

);

Copie

Cette définition de table spécifie les contraintes de clé étrangère pour les champs brand_idet category_id, en précisant qu'elles référencent respectivement la table brandset . categoriesCette définition de table spécifie également que les lignes référencées ne doivent pas être autorisées à être supprimées et, si elles sont mises à jour, les modifications doivent également être reflétées dans les références.

Le attributestype de colonne du champ a été déclaré JSON, qui est le type de données natif désormais disponible dans MySQL. Cela vous permet d'utiliser les différentes constructions liées à JSON dans MySQL sur le attributesterrain.

Voici un diagramme de relation d'entité de la base de données créée :

Cette conception de base de données n'est pas la meilleure en termes d'efficacité et de précision. Il existe des cas d'utilisation courants dans le monde réel qui ne sont pas pris en compte. Par exemple, il n'y a pas de colonne de prix dans le productstableau et il n'y a pas de prise en charge pour un produit appartenant à plusieurs catégories. Cependant, le but de ce tutoriel n'est pas d'enseigner la conception de base de données mais plutôt comment modéliser des objets de nature différente dans une seule table en utilisant les fonctionnalités JSON de MySQL.

Il a également utilisé la JSON_ARRAYfonction qui renvoie un tableau JSON lorsqu'un ensemble de valeurs est passé.

Si vous spécifiez plusieurs fois une même clé, seule la première paire clé/valeur sera conservée. C'est ce qu'on appelle normaliser le JSON dans les termes de MySQL. De plus, dans le cadre de la normalisation, les clés d'objet sont triées et l'espace blanc supplémentaire entre les paires clé/valeur est supprimé.

De plus, vous pouvez utiliser les fonctions intégrées JSON_MERGE_PRESERVEou JSON_MERGE_PATCHpour créer des objets JSON.

Remarque : dans les versions précédentes de MySQL, vous pouviez utiliser JSON_MERGE, mais cette fonction est obsolète.

'JSON_MERGE' is deprecated and will be removed in a future release. Please use JSON_MERGE_PRESERVE/JSON_MERGE_PATCH instead

Étape 3 - Lecture des données du champ JSON

Maintenant que vous avez quelques produits dans la base de données avec lesquels travailler, vous pouvez expérimenter la lecture des données.

Pour les valeurs MySQL typiques qui ne sont pas de type JSON, vous vous fierez généralement à une WHEREclause. Heuristiquement, lorsque vous travaillez avec des colonnes JSON, cela ne fonctionne pas.

Lorsque vous souhaitez sélectionner des lignes à l'aide d'un champ JSON, vous devez être familiarisé avec le concept d'expression de chemin. Les expressions de chemin utilisent un signe dollar ( $) et les clés de l'objet cible.

Lorsqu'il est utilisé en combinaison avec la JSON_EXTRACTfonction, vous pouvez récupérer les valeurs de la colonne spécifiée.

Considérez un scénario où vous êtes intéressé par tous les téléviseurs qui ont au moins un port USB et un port HDMI :

SELECT

        *

FROM

        `e_store`.`products`

WHERE

        `category_id` = 1

AND JSON_EXTRACT(`attributes` , '$.ports.usb') > 0

AND JSON_EXTRACT(`attributes` , '$.ports.hdmi') > 0;

Copie

Le premier argument de la JSON_EXTRACTfonction est le JSON pour appliquer l'expression de chemin à laquelle se trouve la attributescolonne. Le $symbole symbolise l'objet avec lequel travailler. Les expressions $.ports.usbet $.ports.hdmipath se traduisent respectivement par "prendre la clé usb sous les ports" et "prendre la clé hdmi sous les ports".

Une fois que vous avez extrait les clés qui vous intéressent, vous pouvez utiliser les opérateurs MySQL tels que le symbole supérieur à ( >).

Cette requête produira trois résultats :

Ces trois téléviseurs possèdent au moins un port USB et un port HDMI. Les modèles « Bravia » et « Proton » ne remplissent pas ces conditions.

Alternativement, la JSON_EXTRACTfonction a l'alias ->que vous pouvez utiliser pour rendre vos requêtes plus lisibles.

Modifiez la requête précédente pour utiliser l' ->alias :

SELECT

        *

FROM

        `e_store`.`products`

WHERE

        `category_id` = 1

AND `attributes` -> '$.ports.usb' > 0

AND `attributes` -> '$.ports.hdmi' > 0;

Copie

Maintenant, vous pouvez lire les données du champ JSON.

Étape 4 - Mise à jour des données dans le champ JSON

Vous pouvez mettre à jour les données dans les champs JSON avec les fonctions JSON_INSERT, JSON_REPLACEet JSON_SET. Ces fonctions nécessitent également une expression de chemin pour spécifier les parties de l'objet JSON à modifier. La sortie de ces fonctions est un objet JSON valide avec les modifications appliquées.

Tout d'abord, mettez à jour les champs JSON avec JSON_INSERTpour ajouter une nouvelle chipsetclé avec la valeur "Qualcomm" pour tous les téléphones mobiles :

UPDATE `e_store`.`products`

SET `attributes` = JSON_INSERT(

        `attributes` ,

        '$.chipset' ,

        'Qualcomm'

)

WHERE

        `category_id` = 2;

Copie

L' $.chipsetexpression de chemin identifie la position de la chipsetpropriété à la racine de l'objet.

Examinez la catégorie de téléphone mobile mise à jour avec la requête suivante :

SELECT

        *

FROM

        `e_store`.`products`

WHERE

        `category_id` = 2

Copie

« Qualcomm » est désormais présent sur tous les mobiles :

Maintenant, mettez à jour les champs JSON avec JSON_REPLACEpour modifier la chipsetclé existante avec la valeur « Qualcomm Snapsdragon » pour tous les téléphones mobiles :

UPDATE `e_store`.`products`

SET `attributes` = JSON_REPLACE(

        `attributes` ,

)

WHERE

Copie

"Qualcomm" est désormais remplacé par "Qualcomm Snapdragon" pour tous les téléphones mobiles :

Enfin, mettez à jour les champs JSON avec JSON_SETpour ajouter une nouvelle body_colorclé avec la valeur "rouge" pour tous les téléviseurs :

UPDATE `e_store`.`products`

SET `attributes` = JSON_SET(

        `attributes` ,

        '$.body_color' ,

        'red'

la couleur "rouge" est désormais appliquée à tous les téléviseurs :

Toutes ces fonctions semblent identiques mais il y a une différence dans leur comportement.

La JSON_INSERTfonction n'ajoutera la propriété à l'objet que si elle n'existe pas déjà.

La JSON_REPLACEfonction remplace la propriété uniquement si elle est trouvée.

La JSON_SETfonction ajoutera la propriété si elle n'est pas trouvée, sinon remplacez-la.

Maintenant, vous pouvez mettre à jour les données du champ JSON.

Étape 5 - Suppression de données du champ JSON

Vous pouvez supprimer des données dans les champs JSON avec la JSON_REMOVEfonction et DELETE.

JSON_REMOVEvous permet de supprimer une certaine clé/valeur de vos colonnes JSON.

À l'aide JSON_REMOVEde la fonction, il est possible de supprimer les mount_typepaires clé/valeur de toutes les caméras :

UPDATE `e_store`.`products`

SET `attributes` = JSON_REMOVE(`attributes` , '$.mount_type')

WHERE

        `category_id` = 3;

Copie

La JSON_REMOVEfonction renvoie le JSON mis à jour après avoir supprimé la clé spécifiée en fonction de l'expression de chemin.

Vous pouvez également DELETEutiliser des lignes entières à l'aide d'une colonne JSON.

A l'aide DELETEde et JSON_EXTRACTet LIKE, il est possible de supprimer tous les téléphones portables disposant de la version « Jellybean » du système d'exploitation Android :

DELETE FROM `e_store`.`products`

WHERE `category_id` = 2

AND JSON_EXTRACT(`attributes` , '$.os') LIKE '%Jellybean%';

Copie

Cette requête supprimera les modèles de téléphones mobiles "Désir" et "Passion".

Travailler avec un attribut spécifique nécessite l'utilisation de la JSON_EXTRACTfonction. Tout d'abord, la ospropriété des téléphones mobiles est extraite. Et puis l' LIKEopérateur est appliqué à DELETEtous les enregistrements qui contiennent la chaîne Jellybean.

Maintenant, vous pouvez supprimer des données du champ JSON.

Étape 11 - Recherche basée sur les attributs JSON

Vous pouvez également interroger les colonnes JSON à l'aide de l'ORM Eloquent.

Considérez une page de recherche qui permet aux utilisateurs de rechercher des caméras en fonction des attributs qui les intéressent.

public function search(Request $request){

    $cameras = \App\Models\Product::where([

        ['attributes->processor', 'like', $request->processor],

        ['attributes->sensor_type', 'like', $request->sensor_type],

        ['attributes->monitor_type', 'like', $request->monitor_type],

        ['attributes->scanning_system', 'like', $request->scanning_system]

    ])->get();

    return view('product.camera.search', ['cameras' => $cameras]);

}

Copie

Les enregistrements récupérés seront désormais disponibles pour la product.camera.searchvue en tant que $camerascollection. Cela vous permettra de parcourir les résultats et d'afficher les caméras qui satisfont aux conditions de la requête de recherche de l'utilisateur.

Étape 12 - Suppression d'un produit

À l'aide d'un attribut de colonne non JSON, vous pouvez supprimer des produits en spécifiant une whereclause, puis en appelant la deleteméthode.

Par exemple, dans le cas d'un id.

\App\Models\Product::where('id', $id)->delete();

Copie

Pour les colonnes JSON, spécifiez une whereclause à l'aide d'un ou de plusieurs attributs, puis appelez la deleteméthode.

\App\Models\Product::where('attributes->sensor_type', 'CMOS')->delete();

}

Copie

Dans cet exemple, ce code supprimera tous les produits dont l' sensor_typeattribut est défini sur "CMOS".

Conclusion

Dans cet article, vous avez conçu une base de données MySQL avec le type de données JSON et vous y êtes connecté avec une application Web Laravel.

Chaque fois que vous avez besoin d'enregistrer des données sous forme de paires clé/valeur dans une table séparée ou de travailler avec des attributs flexibles pour une entité, vous devez envisager d'utiliser un champ de type de données JSON à la place, car il peut fortement contribuer à compresser la conception de votre base de données.

Si vous souhaitez approfondir, la documentation MySQL est une excellente ressource pour explorer plus avant les concepts JSON.
